## The Four Shifts This Week

Week 12 made four moves. Each one is a habit you carry into every backend you build from now on.

### 1. From SQLite to PostgreSQL

Not because SQLite is bad, but because the rest of the Marathon assumes a real database. Postgres gave you:

- Real types that refuse bad data at write time.
- `TIMESTAMPTZ` instead of date strings.
- `UUID` primary keys generated by the database, not by Node.
- `JSONB` for flexible payloads you can still query.
- `CHECK` constraints, `UNIQUE` constraints, `FOREIGN KEY` with `ON DELETE` actions that Postgres enforces by default.
- A user-and-database model that lets your app connect with a restricted account instead of superuser.

If someone asks you "why Postgres?" on Friday, answer with concurrency first, real types second, `JSONB` third. Not "because the course said so".

### 2. From `better-sqlite3` to `pg` with Pools and Parameters

You learned two things the same day: how to use a connection pool, and how to never, ever, concatenate SQL strings. Parameterised queries are the difference between an app and an app that gets pwned the first time a stranger types `'; DROP TABLE ...` into a form.

Memorise these three bits:

```javascript
// THE pattern
const { rows } = await pool.query("SELECT * FROM leads WHERE id = $1", [id]);

// RETURNING to avoid a second round trip
await pool.query("INSERT INTO leads (wa_phone) VALUES ($1) RETURNING *", [phone]);

// Dynamic WHERE with positional params
const params = [];
const conditions = [];
if (status) { params.push(status); conditions.push(`status = $${params.length}`); }
```

If you can write those three snippets from memory, you can build any read/write path in this course.

### 3. From One Big File to a Layered Codebase

The Week 11 `index.js` had routes, SQL, business rules, and error handling all mixed together. You split it into:

```
routes       know URLs
controllers  know req/res
services     know business rules
repositories know SQL
middleware   is glue
```

The test: if you deleted Express tomorrow, services and repositories should still work unchanged. Only routes and controllers would need replacement.

The payoff of the refactor was visible twice in the same week:

- On Day 3 you added authentication in **one** middleware line and **one** new route file. No change to any controller. If the code had still been a single file, that change would have sprayed `jwt.verify` across every handler.
- On Day 4 you added authorization in the service layer, passing `req.user` as the first argument of each service function. Every rule is one place, visible, testable.

### 4. From "anyone can hit the URL" to Authenticated, Authorized, Multi-User

You now have:

- A `users` table with hashed passwords (bcrypt, 12 rounds).
- A `/api/auth/signup` and `/api/auth/login` that return JWTs.
- A `requireAuth` middleware that protects every lead route.
- A `requireRole("admin")` middleware for admin-only routes.
- Authorization rules in the service that decide *which* leads a given user may see.
- A React login page, `localStorage` token, and a `ProtectedRoute` wrapper.

That is the full authentication-and-authorization story at a level you can take to an interview.

## Self-Review Questions

Answer each of these out loud or in a notebook before the peer session. These are the questions a friend of mine who runs a Nairobi dev team asks in interviews, so pretend it is an interview.

**Postgres**

1. What happens if you run `INSERT INTO leads (wa_phone, name, status) VALUES (...)` and pass a status that is not in the allowed list?
2. What is the difference between `TIMESTAMPTZ` and `TIMESTAMP`?
3. Name three columns on `leads` that have indexes and one reason for each.
4. What does `ON DELETE CASCADE` do? What does `ON DELETE SET NULL` do? When would you pick each?
5. Why is `gen_random_uuid()` preferable to generating UUIDs in Node?
6. What does `RETURNING *` do in an `INSERT` statement, and what problem does it solve?

**Node + `pg`**

7. What is a connection pool and why do we use one?
8. What is `$1`, `$2` in a query string? What would happen if you used string concatenation instead?
9. What happens to an async route handler that throws if it is not wrapped in `asyncHandler`?
10. Why does the error-handling middleware take four arguments instead of three?
11. Which Postgres error code means "unique violation"? "Check constraint violation"?

**Architecture**

12. If you want to add a new business rule -- "a lead must have an email before it can be marked `qualified`" -- which file does it go in?
13. If you want to change the shape of the SQL query behind `listLeads`, which file do you edit?
14. If you want to change the HTTP status code for "lead not found", which file do you edit?
15. Why is `req.user` passed as a first argument to service functions instead of stashed globally?

**Auth**

16. Why do we store `password_hash` and not `password`?
17. Why does bcrypt salt automatically and why does that matter?
18. What is inside a JWT? Which part is secret?
19. If a user's JWT is stolen, what can the attacker do, and how do we limit the damage?
20. Why does our login endpoint return the same error message for "wrong password" and "unknown email"?
21. Why does an agent fetching a lead they do not own receive a 404 instead of a 403?

If you can answer 17 out of 21 clearly, you are ready for the weekend project. Below 15, revisit the day the weakest answers came from.

## Peer Coding Session (Fridays, 3 hours)

Pair with one other student. Each pair picks **one** of the tracks below and spends the session on it. Report back with a two-minute demo at the end.

### Track A: Migrations as code

Right now your schema changes are typed directly into `psql`. That is fine for a week but will break the moment two developers work on the same project. In 90 minutes, install `node-pg-migrate` (`npm install -g node-pg-migrate`), and rewrite `db/schema.sql` as a series of numbered migration files in `db/migrations/`. Goal: `npm run migrate up` on a fresh database gives you the Week 12 schema. Bonus: write a `npm run migrate down` to reverse.

### Track B: Integration tests with a real database

Install `jest` and `supertest`. Write a test that spins up a second Postgres database called `crm_test`, loads `schema.sql`, creates a user, logs in, posts a lead through the webhook (stubbing the Meta call), fetches it back via `/api/leads`, and confirms the shape. Goal: one `npm test` run that touches every layer you built this week. This is the first integration test in the whole course -- doing it now, on a tiny surface, is much easier than adding it later.

### Track C: Authorization edge cases

Together, come up with five ways an agent could *try* to see another agent's leads, and test whether your service rejects each one. Examples: passing another agent's id as a query param, setting `Authorization: Bearer admin-token-they-guessed`, editing one byte of a real token, logging in as one user and using a cookie from another session. For each attack that *works*, write the fix. This track is less code and more mischief -- you will learn more from it than from any article on "API security".

### Track D: Seed and reset scripts

Write `db/seed.sql` that inserts: two users (one admin, one agent), five leads (three assigned to the agent, two unassigned), two conversations, and ten messages. Then write a `npm run db:reset` script that drops every table, re-runs `schema.sql`, and re-runs `seed.sql`. Goal: a fresh dev machine can `git clone`, `npm install`, `npm run db:reset`, `npm run dev`, and see real data in under a minute. This is the single most important thing you can add to the project's developer experience.
